1. Running a Single Test

Execute individual test scenarios and monitor their results

To run a single test, click the "Run Test" button on any test scenario card. The application will execute the test and display results in real-time.

JavaScript Code Example

// Run a single test by ID
async function runTest(testId) {
    const test = appState.tests[testId];

    // Update status to running
    updateTestStatus(testId, 'running');

    try {
        // Simulate API call
        const result = await simulateAPICall(test);

        // Update state with results
        appState.tests[testId] = {
            ...test,
            status: result.success ? 'passed' : 'failed',
            result: result,
            executionTime: result.executionTime
        };

        // Display results
        displayTestResult(testId, result);
    } catch (error) {
        console.error('Test failed:', error);
    }
}

Step-by-Step Walkthrough

  1. Locate Test Card

    Find the test scenario you want to run on the dashboard. Each card displays the test name, description, and current status.

  2. Click Run Test

    Click the "Run Test" button. The button will change to show "Running..." and a loading indicator.

  3. Monitor Progress

    Watch the status badge update from "Pending" to "Running" and finally to either "Passed" or "Failed".

  4. View Results

    Results appear below the test description, including execution time, status code, and response data.

Screenshot: Running a single test scenario

Expected Outcome: The test completes within 1-3 seconds, displaying either a success message with response data or an error message with diagnostic information.

2. Running All Tests

Execute all test scenarios sequentially

The "Run All Tests" feature executes all configured test scenarios in sequence, providing a comprehensive overview of your application's health.

JavaScript Implementation

// Run all tests sequentially
async function runAllTests() {
    if (appState.isRunningAll) return;

    appState.isRunningAll = true;

    // Iterate through all test scenarios
    for (const scenario of TEST_SCENARIOS) {
        await runTest(scenario.id);

        // Small delay for better UX
        await new Promise(resolve => setTimeout(resolve, 300));
    }

    appState.isRunningAll = false;
    console.log('All tests completed');
}

Usage Instructions

  1. Click "Run All Tests"

    Locate the "Run All Tests" button at the top of the dashboard and click it.

  2. Watch Sequential Execution

    Tests run one after another. Each test card updates its status as it executes.

  3. Review Dashboard Statistics

    The dashboard statistics update in real-time showing total tests, passed, failed, and success rate.

Performance Note: Tests run sequentially with a 300ms delay between each test for optimal user experience and to prevent overwhelming the system.

4. Exporting Data

Export test results for analysis and reporting

Export test results to JSON format for further analysis, reporting, or integration with other tools.

Export Function Implementation

// Export results as JSON file
function exportResults() {
    const data = {
        tests: appState.tests,
        history: appState.history,
        exportDate: new Date().toISOString()
    };

    const blob = new Blob(
        [JSON.stringify(data, null, 2)],
        { type: 'application/json' }
    );

    const url = URL.createObjectURL(blob);
    const a = document.createElement('a');
    a.href = url;
    a.download = `test-results-${Date.now()}.json`;
    a.click();
    URL.revokeObjectURL(url);
}

// Call from browser console or button
window.exportResults();

Exported Data Structure

    "tests": {
        "auth-test": {

            "name": "User Authentication Test",
            "status": "passed",
            "executionTime": 1234,
            "timestamp": "2025-11-13T10:30:45.123Z"
        }
    },
    "history": [
        {
            "testId": "auth-test",
            "result": { /* result object */ },
            "timestamp": "2025-11-13T10:30:45.123Z"
        }
    ],
    "exportDate": "2025-11-13T10:31:00.000Z"
}

How to Export

  1. Open Browser Console

    Press F12 or right-click and select "Inspect" to open developer tools.

  2. Run Export Command

    Type exportResults() in the console and press Enter.

  3. Save File

    A JSON file will automatically download with a timestamp in the filename.

Integration Tip: Use exported JSON data with CI/CD pipelines, reporting tools, or custom analysis scripts.

5. Using localStorage

Persistent storage for test history and results

The application automatically saves test history to browser localStorage, allowing results to persist across sessions.

localStorage Implementation

// Save test history to localStorage
function saveTestHistory() {
    try {
        localStorage.setItem(
            'quick-test-history',
            JSON.stringify(appState.history)
        );
    } catch (error) {
        console.error('Failed to save test history:', error);
    }
}

// Load test history from localStorage
function loadTestHistory() {
    try {
        const saved = localStorage.getItem('quick-test-history');
        if (saved) {
            appState.history = JSON.parse(saved);
            console.log(
                'Test history loaded:',
                appState.history.length,
                'entries'
            );
        }
    } catch (error) {
        console.error('Failed to load test history:', error);
    }
}

localStorage Key Reference

Key Data Type Description
quick-test-history Array Stores last 50 test results with timestamps

Managing localStorage Data

// View stored data
console.log(localStorage.getItem('quick-test-history'));

// Clear stored data
localStorage.removeItem('quick-test-history');

// Clear all localStorage
localStorage.clear();
Privacy Note: localStorage data persists in the browser and is accessible to any script on the same domain. Do not store sensitive information.

6. Understanding Test States

Complete lifecycle of test execution

Tests progress through four distinct states during their lifecycle. Understanding these states helps interpret test results and application behavior.

Test State Definitions

State Badge Description Actions Available
Pending PENDING Test has not been executed yet Run Test, Run All Tests
Running RUNNING Test is currently executing None (wait for completion)
Passed PASSED Test completed successfully Run Test again, View Results
Failed FAILED Test completed with errors Run Test again, Debug, View Error

State Transition Diagram

Pending → Running → Passed/Failed

State Management Code

// Update test status and UI
function updateTestStatus(testId, status) {
    const badge = document.getElementById(`badge-${testId}`);
    const card = document.getElementById(`card-${testId}`);

    if (badge) {
        badge.textContent = status.charAt(0).toUpperCase() + status.slice(1);
        badge.className = `test-status-badge status-${status}`;
    }

    if (card) {
        card.className = `test-card status-${status}`;
    }
}

8. Common Use Cases

Practical scenarios and implementation patterns

Use Case 1: Basic Testing Workflow

// 1. Run individual tests during development
await runTest('auth-test');

// 2. Review results and fix issues
const result = appState.tests['auth-test'].result;
if (!result.success) {
    console.error('Test failed:', result.error);
}

// 3. Run all tests before deployment
await runAllTests();

// 4. Export results for documentation
exportResults();

Use Case 2: Automated Testing Setup

// Run tests automatically on page load
document.addEventListener('DOMContentLoaded', async () => {
    // Wait for initialization
    await new Promise(resolve => setTimeout(resolve, 1000));

    // Run all tests
    await runAllTests();

    // Check results
    const failed = Object.values(appState.tests)
        .filter(t => t.status === 'failed');

    if (failed.length > 0) {
        console.error(`${failed.length} tests failed!`);
    } else {
        console.log('All tests passed!');
    }
});

Use Case 3: CI/CD Integration

// Node.js script for CI/CD pipeline
const puppeteer = require('puppeteer');

async function runCITests() {
    const browser = await puppeteer.launch();
    const page = await browser.newPage();

    // Navigate to test app
    await page.goto('http://localhost:3000');

    // Run all tests
    await page.evaluate(async () => {
        await runAllTests();
    });

    // Get results
    const results = await page.evaluate(() => {
        return appState.tests;
    });

    // Check for failures
    const hasFailures = Object.values(results)
        .some(t => t.status === 'failed');

    await browser.close();

    // Exit with appropriate code
    process.exit(hasFailures ? 1 : 0);
}

runCITests();

Use Case 4: Debugging Failed Tests

// Get all failed tests
const failedTests = Object.values(appState.tests)
    .filter(t => t.status === 'failed');

// Log detailed error information
failedTests.forEach(test => {
    console.group(`Failed Test: ${test.name}`);
    console.log('Error:', test.result.error);
    console.log('Status Code:', test.result.status);
    console.log('Execution Time:', test.executionTime, 'ms');
    console.log('Timestamp:', test.timestamp);
    console.groupEnd();
});

// Retry failed tests
for (const test of failedTests) {
    console.log(`Retrying: ${test.name}`);
    await runTest(test.id);
}

9. Best Practices

Testing strategies, performance tips, and common pitfalls

Testing Strategies

Do's

  • Run individual tests during development for faster feedback
  • Run all tests before committing code changes
  • Review test results immediately after execution
  • Export results periodically for historical tracking
  • Use descriptive test names that clearly indicate what is being tested
  • Monitor execution times to identify performance regressions

Don'ts

  • Don't ignore failed tests - investigate and fix immediately
  • Don't run tests in production environments
  • Don't store sensitive data in localStorage
  • Don't modify test state directly - use provided functions
  • Don't run multiple test executions simultaneously

Performance Tips

1. Optimize Test Execution

// Bad: Running tests without waiting
runTest('test1');
runTest('test2'); // May cause conflicts

// Good: Wait for test completion
await runTest('test1');
await runTest('test2');

2. Manage localStorage Size

// Clear old history periodically
if (appState.history.length > 50) {
    appState.history = appState.history.slice(0, 50);
    saveTestHistory();
}

3. Monitor Test Duration

// Set up performance monitoring
const monitorPerformance = () => {
    const slowTests = Object.values(appState.tests)
        .filter(t => t.executionTime > 3000);

    if (slowTests.length > 0) {
        console.warn('Slow tests detected:', slowTests);
    }
};

Common Pitfalls to Avoid

Pitfall Impact Solution
Not waiting for async operations Race conditions, unreliable results Always use await with test functions
Modifying appState directly UI not updating, inconsistent state Use provided state management functions
Ignoring execution time increases Performance degradation over time Monitor and investigate slow tests
Not clearing localStorage Storage quota exceeded Implement periodic cleanup or limits
Running tests in parallel Resource conflicts, invalid results Use sequential execution with delays

Code Quality Guidelines

Maintainable Test Code

// Good: Clear, documented, error-handled
async function customTestWorkflow() {
    try {
        // 1. Reset state
        resetTests();

        // 2. Run critical tests
        await runTest('auth-test');
        await runTest('db-test');

        // 3. Check results
        const criticalPassed = ['auth-test', 'db-test']
            .every(id => appState.tests[id].status === 'passed');

        if (criticalPassed) {
            console.log('Critical tests passed');
            // 4. Run remaining tests
            await runAllTests();
        }
    } catch (error) {
        console.error('Workflow failed:', error);
    }
}

Security Considerations

  • Never store authentication tokens in localStorage
  • Avoid logging sensitive data to console
  • Sanitize test inputs to prevent injection attacks
  • Use HTTPS for all API endpoint tests
  • Implement rate limiting for test execution
